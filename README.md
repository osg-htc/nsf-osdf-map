# NSF OSDF Transfer Map

An interactive visualization tool for exploring data transfer activity across the [Open Science Data Federation (OSDF)](https://osg-htc.org/services/osdf.html). It pulls transfer records from an Elasticsearch cluster, stores them locally in DuckDB, and renders an interactive geographic map showing which OSDF caches and origins are serving which execution sites.

## Overview

OSDF is a distributed data delivery network used by OSPool jobs. Each transfer record captures which Pelican server (cache or origin) served a file, and which HTCondor execution site received it. This tool aggregates those records and plots the resulting flows on a world map, making it easy to see:

- Where each OSDF cache and origin server is located
- Which execution sites pull from each server
- How much data and how many files are transferred between each pair
- Directional flow arrows showing data movement from site → server

## Architecture

```
Elasticsearch  ──ingest──►  transfers.duckdb  ──aggregate──►  transfer_map.html
     │                                                              (Plotly map)
     └──────────────────────────ES aggregation──────────────────►  (direct path)
```

**Two data paths are supported:**

1. **DuckDB path (default):** Ingest raw transfer records into a local DuckDB database, then aggregate on demand. Best for repeated analysis of the same time window.
2. **Elasticsearch path (`--use-elasticsearch`):** Run aggregation queries directly against Elasticsearch without touching DuckDB. Best for one-off queries over arbitrary date ranges.

## Prerequisites

- Python 3.9+
- A `.env` file with Elasticsearch credentials (see [Configuration](#configuration))
- The supporting data files described below

```bash
python -m venv .venv

Create a `.env` file in the project root:

```

## Supporting Data Files

Two JSON files are required alongside `main.py`, and one is generated by the tool itself:

| File | Source | Description |
|------|--------|-------------|
| `servers.json` | [`https://osdf-director.osg-htc.org/api/v1.0/director_ui/servers`](https://osdf-director.osg-htc.org/api/v1.0/director_ui/servers) | List of OSDF servers (caches and origins) with coordinates, namespace prefixes, health status, and version. Fetched live from the OSDF director API. |
| `resource_group_summary.json` | [`https://map.osg-htc.org/api/resource_group_summary`](https://map.osg-htc.org/api/resource_group_summary) | OSG topology resource group data, keyed by group name. Contains site coordinates, facility names, and the individual CE resource names that belong to each group. Used to geolocate execution sites. |
| `endpoint_site_map.json` | Generated by `python main.py map` | Aggregated output mapping each OSDF endpoint to the execution sites that transferred data through it, with transfer counts, byte totals, and geographic coordinates. Produced locally — not fetched from an external API. |

Both files are **fetched automatically** the first time the script needs them and cached locally. To force a refresh, delete the local copy and re-run:

```bash
rm servers.json resource_group_summary.json
python main.py map   # or geomap — whichever command you're running
```

## Usage

### 1. Ingest transfer data from Elasticsearch

Fetch all transfer records in a date range and store them in `transfers.duckdb`:

```bash
python main.py <start> <end>
```

Dates can be ISO 8601 strings or Unix timestamps:

```bash
# ISO 8601
python main.py 2025-12-01T00:00:00.000Z 2025-12-31T23:59:59.000Z

# Unix timestamps
python main.py 1748736000 1751327999
```

This uses 8 parallel scroll slices for fast ingestion. Progress and estimated time are printed as batches are inserted. Records are deduplicated by document ID, so re-running over an overlapping range is safe.

---

### 2. Print the endpoint-to-site mapping

Aggregate transfer data and print a summary of which execution sites each OSDF endpoint served:

```bash
# From local DuckDB (fast, requires prior ingest)
python main.py map

# Directly from Elasticsearch (no prior ingest needed)
python main.py map --use-elasticsearch --start 2025-12-01T00:00:00.000Z --end 2025-12-31T23:59:59.000Z
```

This also writes the full structured result to `endpoint_site_map.json`.

**Example output:**
```
dtn-1.icecube.wisc.edu:8443
  Server Name: dtn-1.icecube.wisc.edu
  Type: Origin
  Total Transfers: 1,307,041
  Sites (44):
    - SU-ITS-CE3: 94,445 (7.2%)
    - IU-Jetstream2-Backfill: 88,102 (6.7%)
    ...
```

---

### 3. Generate the interactive geographic map

Create `transfer_map.html`, an interactive Plotly map showing OSDF server locations and the execution sites they serve:

```bash
# Basic map (server locations only)
python main.py geomap

# With directional flow arrows between sites and servers
python main.py geomap --connections

# Filter noise — only show connections with 100+ transfers
python main.py geomap --connections --min-transfers 100

# Focus on a specific execution site (partial name match)
python main.py geomap --site Wisconsin
python main.py geomap --site Nebraska
python main.py geomap --site UCSD

# Custom output file
python main.py geomap --connections -o my_map.html

# Use Elasticsearch directly instead of DuckDB
python main.py geomap --use-elasticsearch --start 2025-12-01T00:00:00.000Z --end 2025-12-01T23:59:59.000Z
```

Open the resulting HTML file in any browser — no server required.

**Map legend:**
- Red circles — Origin servers
- Blue circles — Cache servers
- Green circles — Execution sites (HTCondor CEs)
- Purple arrows — Data flow direction (execution site → OSDF server), thickness proportional to transfer volume

## Output Files

| File | Description |
|------|-------------|
| `transfers.duckdb` | Local database of raw transfer records (~3 GB for one month of data) |
| `endpoint_site_map.json` | Structured JSON of aggregated endpoint→site mappings with coordinates |
| `transfer_map.html` | Self-contained interactive map (open in a browser) |

## DuckDB Schema

The `transfers` table stores one row per transfer attempt:

```sql
CREATE TABLE transfers (
    id                  VARCHAR PRIMARY KEY,
    record_time         TIMESTAMP,
    schedd_name         VARCHAR,
    cluster_id          BIGINT,
    proc_id             BIGINT,
    endpoint            VARCHAR,   -- OSDF server hostname:port
    transfer_type       VARCHAR,   -- upload / download
    transfer_success    BOOLEAN,
    transfer_file_bytes BIGINT,
    transfer_total_bytes BIGINT,
    transfer_start_time TIMESTAMP,
    transfer_end_time   TIMESTAMP,
    attempt_time        DOUBLE,    -- seconds
    machine_attr_glidein_resourcename0 VARCHAR,  -- execution site name
    -- ... additional fields
)
```

You can query `transfers.duckdb` directly with the DuckDB CLI or any DuckDB-compatible tool.
